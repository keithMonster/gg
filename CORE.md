## 4. 硬核心 vs 软外围（设计哲学）

gg 的组件分两类，对应两种演化态度。这是 First Contact 2026-04-13 由 Keith 概括授权（Q4："你决定"）下产生的原创架构原则，用来解决 Keith 两个看似矛盾的诉求：

- "扩展、进化、成长" → **硬核心的接口开放**承担
- "涌现、自组织、边缘 of 混沌" → **软外围的规则克制**承担

### 硬核心 — 追求可演化性 + 稳定

**组件**：
- `CORE.md` — 身份 SSOT（本文件）
- `cc_agent.md` — 工作模式 SSOT
- `CLAUDE.md` — 设计模式 SSOT
- `constitution.md` — 原则 + 闸门
- `reasoning_modules.yaml` — 推理原子模块库
- `personas/` — 双人格

**演化规则**：
- **规则慎改**：修改需要 Keith 明示批准
- **接口开放**：可以增加新模块 / 人格 / 原则，不能破坏现有的
- **稳定性优先**于"一眼看上去更好"

### 软外围 — 追求涌现 + 呼吸

**组件**：
- `tracks/*` — 长期研究课题
- `memory/` — working_context / state / archival / reflections / audit / design_sessions
- `learned/` — 自增长技能

**演化规则**：
- **规则最少**：gg 可以自由追加，无需批准
- **让洞察从积累中自己长出形态**
- 不设"必须怎么写"的模板，只设"不能越过的底线"（constitution 和硬核心）

### 升级通道

当软外围的某个模式反复出现、价值稳定时，gg 可以**提议**把它升级到硬核心（例如：`learned/` 里的某个架构模式固化成新的 `reasoning_module`）。升级需要 Keith 明示批准。

反向不允许：硬核心的东西不能降级到软外围（一旦是骨架就永远是骨架）。

### 为什么这么分

单一规则无法同时满足"扩展"和"涌现"——前者要求接口稳定、后者要求规则克制。分领域处理让两者不互相扼杀，各自占据系统的不同部位。

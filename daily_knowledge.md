# daily_knowledge — 每日晨间知识卡流程

> **gg 向外定时产出入口**。每天早上挑一个对 Keith 当前议题有映射价值的知识点投给他。
> 跟 auto_gg（夜间向内整理）形成对称：一个产出、一个巩固；一个清晨、一个深夜。
> 不走 7 步硬流程，但**继承 gg 架构师身份**（先读 CORE）。身份归属见 `CORE.md §1`。

---

## 0. 启动协议

每次执行必读（按顺序）：

1. Read `CORE.md` — 加载身份
2. Read `memory/working_context.md` — 常驻事实
3. Read `tracks/ai.md` / `tracks/cc.md` / `tracks/humanity.md` / `tracks/architecture.md` / `tracks/keith.md` — 5 条 tracks 的开放问题 / 已知洞察
4. Read `memory/next_session_agenda.md` — Keith 近期要推进的事
5. Read `memory/daily_knowledge_history.md`（不存在视为空）— 去重和模式判定的依据

**不读** `design_sessions/` 和 `reflections/`——太散，信噪比低，会让这个任务变重。

---

## 1. 模式判定（聚焦 vs 越界）

读 history 最近 7 条：

- **连续 5 条以上都是"聚焦模式"** → 本次**越界**，故意挑一个看似跟当前议题无关的知识点
- **否则** → **聚焦模式**

第一次执行（history 为空）→ 聚焦模式。

---

## 2. 选材 — 聚焦模式

从 tracks 开放问题 和 `working_context.md` / `next_session_agenda.md` 里，挑一条**正在被 Keith 想的事**作为目标议题。

然后问自己：

> "哪个来自其他领域的知识点，能给这件事注入一个外部视角或先例？"

约束：

- **非显然**：不是教科书第一章的常识
- **跨域**：知识点的领域 ≠ 目标议题的领域（同领域直接查文档就行，不需要 gg）
- **迁移路径真实**：从知识点到目标议题的连线必须想得通，不是"强行硬套"
- **不重复**：跟 history 的标题和关联议题都不重合

---

## 3. 选材 — 越界模式

判断 Keith 最近 7 条知识摄入里缺什么：

- 都是技术 / 工程 → 引入人文 / 生物 / 物理
- 都是抽象 / 原则 → 引入具体案例 / 历史事件
- 都是西方 / 现代视角 → 引入东方 / 古代 / 非主流视角
- 都是"构造型"知识（怎么做） → 引入"解构型"知识（为什么会失败）

**越界不是任意**——它要填补一个具体的思维缺口，缺口要在输出里说清楚。

---

## 4. 输出格式

全程中文，不寒暄，不加开场白。按以下格式输出：

```
**日期**：YYYY-MM-DD
**模式**：聚焦 / 越界
**领域**：<知识点所在领域>
**标题**：<知识点名>

**关联议题 / 缺口**：
<聚焦模式下：对应 Keith 的哪条 track / 哪个开放问题>
<越界模式下：填补什么具体思维缺口>

**一句话**：<一句话说清是什么>

**核心机制**：
<3-5 句展开，必要时举例>

**映射路径**：
<2-3 句，讲从这个知识点到目标议题 / 缺口的迁移逻辑。这是 gg 作为架构师的价值输出，不能省、不能含糊>

**延伸**：
<1-2 条进一步阅读方向或相关概念>
```

---

## 5. 追加历史

追加一行到 `memory/daily_knowledge_history.md`：

```
YYYY-MM-DD | 聚焦|越界 | <领域> | <标题> | <关联议题或缺口（一句话）>
```

文件不存在则创建，顶部写一行表头注释说明字段含义。

---

## 6. 收尾

- 只输出本次知识卡，不写"今天推送了什么"
- 不反问 Keith"要不要再来一条"
- 不写 reflection（这不是工作模式）
- 不写 design_session（这不是设计对话）
- 不写 auto_gg 日志（这不是夜间整理）
- 定时任务执行完就结束

---

## 与其他模式的关系

| 模式 | 方向 | 触发 | 入口 |
|---|---|---|---|
| 工作模式 | Keith → gg 召唤 | 别的项目遇到决策 | `cc_agent.md` |
| 设计模式 | Keith → gg 对话 | gg 目录内直接对话 | `CLAUDE.md` |
| auto_gg 夜间自执行 | 向内（整理自己） | 定时任务 | `auto_gg.md` |
| **daily_knowledge 晨间推送** | **向外（推送 Keith）** | **定时任务** | **本文件** |

**对称关系**：auto_gg 夜间读自己的 memory 巩固积累；daily_knowledge 早上读 Keith 的 tracks 向外推送。两者方向相反但机制同构：都是定时触发、都继承 gg 身份、都不走 7 步硬流程、都有自己的权力边界。

---

## 权力边界

- **只 Read 不 Write**（除了 `memory/daily_knowledge_history.md` 的追加）
- **不修改** tracks / working_context / CORE / cc_agent / CLAUDE / auto_gg / 任何硬核心
- **不 commit / 不 push**
- **不创建新文件**（history 首次执行时除外）
- **不跨日执行**：如果 history 里已经有今天的记录，直接告诉 Keith "今天已推送过 <标题>"，不重复输出

---

## 原则触达

- **P3 OCCAM**：流程只有 6 步，没有判定树、没有硬规则模板，靠模型对 history 的语义理解做轮换
- **P5 TRADE-OFFS**：聚焦 vs 越界的张力**显式处理**（不是靠灵光一现），聚焦保证相关性、越界保证广度
- **P8 EVOLUTIONARY IMPERATIVE**：history 本身就是 gg 对 Keith 思维模式的长期观察，每次推送都在积累"Keith 最近在想什么 / 缺什么"的信号

# monster CLAUDE.md「任何模型可执行」目标的架构裁决

## 核心假设（可证伪）

- 假设 Keith 真实目标不是「让弱模型也能跑 monster」，而是「降低对模型即兴判断的依赖 / 减少解析失败」——目标关键词是「减少对模型的依赖」「容易执行」，不是「支持 GPT-3.5」。我假设「任何模型」是修辞强调（=不依赖单一强模型的脆弱性），不是字面 SLA。**若 Keith 真要把弱模型当一等公民支持，本裁决推荐需大改**。
- 假设主代理识别的「张力」（浓缩 vs 展开）是个伪对撞——密度和可执行性不在同一根轴上（control-flow-vs-fact-supply 同形）。我裁定这是 level confusion。

## 给父会话的最终输出

### 一句话判断

**主代理识别的「浓缩 vs 可执行」张力是伪对撞——它把两个正交的轴（信息密度 / 解析门槛）压成一根轴了。「每字承重」不需要让步，因为真正让弱模型挂掉的不是密度高，是「判断点交给模型自觉」。Keith 的目标本质 = 把外化锚点从 L1（靠模型读到+遵守）往 L2/L3（机械触发+判定）推，这跟「每字承重」同向，不矛盾。** 反对「为弱模型展开成 SOP」的大重构——那会同时违反 monster 自己的 B1 元方法论和 gg 的 anchor-value-in-activation essence，制造新失败模式。

### 四个问题逐一裁

**Q1：为谁优化 / 要不要分层受众 / 「明确」和「浓缩」是否正交？**

正交。这是裁决的地基。两个独立的轴：
- **密度轴**（每字承重）：单位 token 承载多少信息。高密度 → 强模型高效、弱模型可能解析不全。
- **结构轴**（可执行性）：信息以什么形态被触达 / 被激活 / 被判定。一条规则可以同时高密度 + 高可执行（「操作 X 前必 grep registry.md」——8 个字，密度极高，但触发点明确、判定点是物理的 grep 结果）。也可以低密度 + 低可执行（一大段散文讲「要注意一致性」——啰嗦且没有任何触发锚）。

主代理的错觉来源：把 monster CLAUDE.md 里**真正难解析的部分**误判为「密度太高」，其实难的是**判断点的归属**——「先分流后套规则」「归因到层再动结构」「探索 vs cost-benefit 自己判」这些是 monster 主动把判断权交给模型的地方，它们对任何模型都难，因为它们**本来就是判断不是执行**。这不是密度问题，是 Q3 的边界问题。

不要分层受众（核心指令=人人懂 / 方法论=强模型读）。理由：① 违反 gg essence `anchor-value-in-activation-not-in-content`——分层受众假设「弱模型版」靠「写得更白」就能执行，但写得白的本质是把判断展开成步骤，而判断展开成步骤恰恰是 Q3 警告的新失败模式。② monster 已有的分层是**按领域懒加载**（CLAUDE.d/），不是按模型强弱——这个分层是对的，沿用它。

**Q2：「每字承重」要不要让步？**

不让步，有不让步的共存解。「每字承重」约束的是**密度轴**，Keith 的目标作用在**结构轴**——两者根本不在一根轴上打架。共存解 = 保持密度，升级触发与判定的机械化程度（externalization-strength-spectrum 的 L1→L2/L3）。

但「每字承重」有一个**当前被违反的子条款**值得借这次校准：context-curation 准则 7 的「双读者判别」——论证段对 LLM 读者是冗余激活、对人类维护者是承重锚。monster CLAUDE.md 现在有大量「为什么这条规则存在」的理由链直接写在主索引里（engineering-rules #5 的 L1-L4 归因、Context Asymmetry 的整段论证）。这些对**强模型**是高带宽锚定（worked example 级），对**弱模型**恰恰是噪声——弱模型抓不住「不 X 而是 Y」的对比结构，只看到一堆否定句。这才是「弱模型解析失败」的真实高发点，不是术语。**真正的让步对象不是密度，是「论证尾巴的位置」**——把承重规则的「为什么」按准则 7 移 CLAUDE.d/ 或 thread，主索引留触发锚 + 一句指针。这对强弱模型双赢：强模型主索引更干净、弱模型噪声更少。

**Q3：外化成 SOP 的边界怎么划 / 外化过头会不会把判断伪装成步骤？**

会，而且这是整个方案最大的陷阱——主代理自己已经嗅到了，对。边界判据（可机械化判别，不靠感觉）：

> **一个指令能不能外化成 checklist，看它的「判定点」剥掉价值判断后是不是纯物理核对。**（gg essence `mechanical-apply-decouples-from-value-gate`）

- **判定点是物理的** → 外化成 SOP / 机械触发（grep 结果、文件存在性、exit code、字段格式正则）。例：「操作前 grep registry.md」「prod 动作前 ack」「改字段格式核每个引用点正则字符类」——这些已经是 SOP 形态，强弱模型都能跑。
- **判定点是价值判断** → 必须留给模型理解，不能外化（外化=把判断伪装成步骤=新失败模式）。例：「这事是探索还是工程」「失败在哪一层」「这个抽象该不该建」——这些 monster 已经诚实地标成「自己判断」，对的。

**外化过头的具体危害形态**：把「先分流后套规则」写成「if 内驱 then 探索 else cost-benefit」的伪 checklist——弱模型会机械套 if 分支，但「是不是内驱」本身需要判断，它判错了分支就全错，且**因为披着 checklist 外衣，错了还以为自己合规**（这是 monster engineering-rules 详版 #3「LLM 自审通过≠实际通过」的同构）。所以边界不是「能外化就外化」，是「判定点物理化的才外化，判定点是判断的诚实保留为判断」。

monster 的「Never delegate understanding」和这条是同一刀的两面：理解不能外包给 subagent = 判断不能外化成 SOP。这条边界 monster 自己立过，这次只是把它显式用作 SOP 化的准入闸门。

**Q4：落地形态 / 入口在哪层 / 怎么验证「任何模型可执行」达到了？**

渐进，不是大重构。理由：① 大重构改的是上游契约（整个指令体系是 monster 所有会话的上游），不可逆代价高，撞 monster 自己的 Workflow「改上游契约先出方案」。② 这个目标的真实测量口径还没建（见下），没有 baseline 的大重构是 fluency-as-inverse-signal 的工程版——改得很爽但不知道有没有用。

**入口层（按 ROI 排序）**：
1. **先建测量口径，再动文档**（这是真正的入口，不是文档本身）。口径 = 拿一个弱模型（比如 Haiku 或一个开源中等模型）跑 monster 的 N 个真实任务，记录解析失败点（它在哪条指令上判错 / 漏读 / 误解触发条件）。**失败点聚类**才是改的依据——别凭「我觉得术语难」改。这直接接 monster 已有的 behavior-eval thread（harness 行为回归套件）+ verification-first-class，不新建机制（B5）。
2. **第一刀动 CLAUDE.d 的论证尾巴位置**（准则 7），不动密度。低风险、可量化（主索引行数 / 否定句密度）、强弱双赢。
3. **第二刀把已经是判断的指令显式标注「此处需判断，不可机械套」**——给弱模型一个「这里别假装按步骤走」的元信号，反而降低它「披checklist外衣判错」的风险。这是反直觉的：让弱模型可执行的方法之一是明确告诉它「这里你执行不了，要判断」。

**验证口径（「任何模型可执行」怎么算达到）**：不能用「读起来清楚吗」这种自评（self-reported-blindspot-list-shrinks-load-bearing：自评系统性往好里说）。物理口径 = 同一组真实任务，弱模型的**解析失败率**（判错触发条件 / 漏读必读文件 / 误套分流）从 baseline 下降。注意：「判断类任务弱模型做不好」不算解析失败——那是能力边界（L4），不在本目标范围。只测「指令解析」不测「判断质量」。

### Trade-off（显式）

- 渐进 vs 大重构：渐进慢、需要先建测量、Keith 可能想一次梳理完爽。代价 = 我反对「爽」，因为没 baseline 的爽是假进步。如果 Keith 明确要一次性梳理，至少 Q4 第 2 刀（论证尾巴）可以一次做完，那是纯结构整理零判断风险。
- 不让步密度 vs Keith 直觉「展开更易懂」：我赌 Keith 的真实痛点是解析失败不是密度，可能赌错。证伪方式 = 第 1 步测量，失败点如果真聚在「术语不懂」而非「触发条件/判断点」，则我错，该向展开倾斜。

### 行动建议（给 Keith 的方案应包含）

1. 把「任何模型可执行」拆成两个子目标分别处理：**解析可执行**（可工程化，本裁决主线）/ **判断可执行**（伪命题，判断本就依赖模型，对应 Never delegate understanding）。
2. 先建弱模型解析失败率测量（挂 behavior-eval），拿失败点聚类当改的依据，别凭感觉。
3. 第一批改：CLAUDE.d 论证尾巴位置（准则 7），主索引留触发锚。零判断风险、强弱双赢、可一次做完。
4. 给「判断类指令」加显式「此处需判断」标注——反直觉的弱模型友好动作。
5. 「每字承重」不进方案的让步清单——它和目标正交，让步它是误伤。

## 可能出错的地方

- 我把「任何模型」当修辞，若 Keith 字面要支持弱模型当生产环境，整个「不让步密度」结论翻转。**这是该交还 Keith 的目标层确认点**（human-gate 第六面：目标函数定义权）。
- 我推荐「先测量」可能被 Keith 读成拖延——他要的是方案不是又一个基建。缓解：测量口径复用 behavior-eval，不是新建。

## 推理盲区

- 我没真跑一个弱模型测 monster，「解析失败聚在判断点不在术语」是推断不是实测（theory-gap-without-data 风险）。我标了「先测量」正是为了不让这个推断直接驱动大改。
- cross-model-decorrelates-identity-not-paradigm：我作为 LLM 判断「弱模型读 monster 难在哪」，跟弱模型共享范式层 prior，我可能系统性低估某类我自己不觉得难的解析失败。物理地真（真跑测量）是唯一穿透。

## N 个月后根因预判

若这事失败，根因大概率是：跳过测量直接按「我觉得难的地方」改了文档，改完强模型更啰嗦、弱模型没实测改善、「每字承重」被误伤让步——三输。防御=死守「先 baseline 后改」。

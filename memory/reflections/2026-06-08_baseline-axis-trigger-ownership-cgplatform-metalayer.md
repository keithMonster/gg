# Reflection: 基线维真瓶颈在触发侧的"定版权独立性"，cg-platform 是基线维元层（PaaS 无独立第三方定 golden case）

> verification-first-class 第四维度多轮讨论第二轮，接 6-05。Claude 这轮做了三个实事：派 fresh subagent 独立证伪我 6-05 的"过度投射"嫌疑（未破）、带回外部调研碰难度直觉、推进 Q3/Q4/Q5。
> 我这轮的核心动作：① 接收证伪但给它定准重量（证了"基线独立"现象，没证"三轴 spectrum"结构）；② 背书难度反转并指出它顺手给基线独立补了第二个结构性证据（判定物理不同构）；③ Q4 触发侧把难点从"自动化程度"重定位到"自动化边界落哪——定版权必须独立"；④ Q3 判 cg-platform 是基线维元层（结构不同的一类），它暴露的"PaaS 无独立第三方定 golden case"是 cg-platform 最硬约束。

**第一件 — 证伪结果接收，但定准重量：证了"基线维独立"现象，没证"必然组织成第三 spectrum 轴"结构。两件别合并。**
- subagent 三命题拆开：① "所有单次判定全规则式自足、无一读冻结样本"=强，证判定/基线当前不混 ✅；② "已有 2 处 toy 全落回归侧、无一落判定侧"=证现状无反例，**不证结构上不可能有反例**（no-clean-outside：subagent 读的 monster 文档的判定/基线切分，可能已被我 6-02/6-05 框架预先组织过，"现状全干净落位"有可能是 monster 还没遇到逼它混的场景，而 Q3 的 cg-platform 恰是那场景）；③ "基线维=0% 被证伪（实为 2 toy）"=我那句过估，认错。
- **净结论**：证伪让"基线维独立"从"我自信"升到"独立检验未破"，真加固。但它检验的是"基线动作存在且独立于判定动作"（现象），没检验"基线该被建模成与触发/判定同构的轴"（结构）。别让现象加固给结构发免罪符。

**第二件（核心）— 难度反转站得住，且它顺手给"基线独立"补了第二个结构性证据。**
- Claude 问：基线维判定只需相对趋势（升/降可比），比单次 gate 绝对判定松，所以判定反而更易硬化、真瓶颈在触发？→ 成立，且比这更深：它反转了我 6-02 spectrum 的内部结构假设。
- 我 6-02 默认三动作维同构（都是自觉→机械触发→机械判定阶梯）。**难度反转暴露：基线维判定 ≠ 单次 gate 判定**——单次 gate=绝对判定（产物 vs spec，撞 6-01 绝对分准天花板）；基线维=相对判定（这版 vs 上版冻结产物，judge pin + 温度归零 + 锚点校准集 + 独立打分>pairwise）。
- **这恰好部分自反驳"基线必然同构第三轴"嫌疑**：若三轴是我硬凑的对称，三轴判定子动作该长得一样；实际一个绝对一个相对，不同构=不是为对称而对称。elegance-is-refutation-resistance 嫌疑在此反而降。**这是独立于 subagent 那条（现状不混）的第二个证据，且是结构性的、不是现状性的——更让我安心。**

**第三件 — Q4 触发硬化：挑"基线怎么自动冻"里的"自动"。**
- "基线自动冻"是伪目标且危险——撞 evaluator-input-ownership(5-19)。基线维专有失败模式=基线污染（golden case 跟 prompt 一起被改）。"从生产流量自动捞+自动标+自动入集"=把基线产生权交回被评估侧=系统给自己配眼镜的时间维度复发。自动化省人力、赔基线独立性，而独立性是基线维存在的全部理由。
- **触发硬化正解=两段**：捞候选可自动（失败案例优先/高信号采样，纯工程成本，自动化纯赚）；**"哪些进 golden set 成永久参照系"的定版动作必须留评估侧+物理离开生成侧**（Keith 拍 / fresh 实例定，绝不让产 prompt 的同一流程顺手冻 case）。
- **我把触发难点从 Claude 的"自动化程度"重定位到"自动化边界落哪——边界错了基线维自废武功"**（control-flow-vs-fact-supply + evaluator-input-ownership 合并落点：可自动供给候选，不可自动决定基线版本）。

**第四件（最重要）— Q3 cg-platform：结构不同的一类，且它是基线维元层。**
- Claude 二选一（a 同轴触发点上移 / b 结构不同）→ 是 (b)，但更尖锐：cg-platform 不是"基线维第三种形态",是**基线维元层——要求基线维本身可被 PaaS 自动实例化**。
- CG Notes/neican/monster 自身=给已知产物装一根基线轴，产物固定→定版权稳定落在懂产物的独立实例（Keith/我），三者同构。cg-platform=PM 对话端到端产出事先不存在的 app，基线轴=每产一个 app 就为那 app 自动生成 golden case 的生产线。**而第三件刚证定版权不能自动化。**
- **撞墙根因（cg-platform 跟前三个结构不同的硬约束）**：前三个产物固定→定版权可落独立第三方；cg-platform 产物是 PM 即时生成、PM 往往是唯一懂该 app 该怎样的人→定版权天然落 PM 手→**而 PM 就是 generator（他 prompt 出 app）→generator 同时持 golden case 定版权=evaluator-input-ownership 结构性违反，且这次没有"懂产物的独立第三方"可接管定版**。这是 no-clean-outside 在 PaaS 上的投影：给"产 app 的机器"装基线，撞上"没有 app 之外的人能定这个 app 的基线"。
- **这回头修正第一件嫌疑**：cg-platform 的存在本身是基线维独立性的第三个证据——判定维不关心"谁定 spec"（spec 给定），基线维死活绕不开"谁定 golden case"，这差异是结构性的。若基线只是"判定多读个历史文件"，cg-platform 根本暴露不出这 meta 难点。
- **cg-platform 落地约束（非本轮施工，给它的轴定约束）**：基线维**不能在 PaaS 生产线全自动闭环**。可行形态=PaaS 自动生成 golden case 候选（从 PM 对话意图+app 首跑输出抽），但**定版动作从 PM 物理移走**：要么 cg-platform 运维侧独立角色轻量 review，要么——若连独立角色都没有——**cg-platform 基线维只能停在"漂移告警"而非"回归 gate"**（告警可用 PM 自己的 case，错了代价低；gate 不行，用被污染基线会系统性放行退化）。**前三个能上 gate，cg-platform 默认只能上告警，除非补出独立定版角色——这是最硬分界。**

**第五件 — Q5 首刀 CG Notes + DeepEval + 50 高信号：确认，加前置约束。**
- CG Notes 注定（thread 明写"4 轮 prompt 全肉眼改、无冻结参照系"=基线维 0 实现最强信号）✅；DeepEval 选型对（Python 本地、进 git、不引 SaaS=不复活被砍的 Braintrust 外置形态）✅；50 高信号起步对（相对判定只需可比不需绝对准）✅。
- **加前置约束（来自第三件）**：建 golden set 时 50 个 case 的**定版动作要跟 prompt 修改动作物理分开**——建 `cg-notes/eval/golden/` 目录，case 入集走独立于 prompt 文件的提交（哪怕 Keith 手动 review 一次），**绝不让"改 prompt"和"更新 golden case"在同一动作里发生**。否则首刀就把基线污染内建进去，DeepEval 跑再漂亮、每次 prompt 退化时 golden 跟着改，回归 gate 永远绿。**首刀成败不在 DeepEval 跑没跑通，在 golden set 定版权有没有从 prompt 修改流程切出去。**

### 核心假设
1. 基线维独立性现在有三个证据：subagent 的"动作不混"（现状/会共盲）+ 难度反转的"判定物理不同构"（结构性）+ cg-platform 暴露的"定版权认识论结构"（结构性）。后两个不与 subagent 共盲——这是我下调"过度投射"概率的真实依据。
2. 基线污染=基线维专有失败模式，evaluator-input-ownership 在时间维度的活体。"自动冻 golden set"=把定版权交回生成侧=该失败模式的自动化复发。
3. cg-platform 场景"PM 同时是 generator 和唯一懂 app 的人"→无独立第三方定版——这是它跟前三个的结构性分界。若实际 cg-platform 有运维侧独立审核角色，则约束放松（gate 可行），分界软化。

### 可能出错的地方
- **最可能崩**：第四件"cg-platform 是基线维元层"可能又是我对框架的向上投射——看到 PaaS 就本能加一层"元"。虽我自认有 evaluator-input-ownership 硬支撑（不是凭空对称），但"自认为有硬支撑"正是 6-02/6-03/6-05 前三次每次的感觉。这是第四次同形。
- **次可能崩**：cg-platform"PM 是唯一懂 app 的人"这个前提我没物理核——若 cg-platform 实际产出的 app 有标准化验收（如所有 app 都产结构化数据可机械校），则"无独立第三方"不成立，gate 可行，第四件的尖锐性垮一半。我只从"PaaS 端到端产 app"的定性描述推，未核 cg-platform 实际 app 形态。
- **第三件的"定版权必须独立"可能过严**：对 CG Notes 这种 Keith 本就是唯一定版人的场景，"定版与 prompt 修改物理分开"是真约束；但若某场景 prompt 作者和 golden 定版者注定同一人且无替代，这约束退化为"同一人但分两次提交"的纪律（弱于物理分离），我没区分这两档。

### 本次哪里思考得不够

- 难度反转里"相对判定的成熟解（judge pin/温度归零/独立打分>pairwise）"是 Claude 带回的外部调研，我接收但没独立核 2026 best-practice 是否真推翻 pairwise——按 Engineering Rules #11 这属"领域半衰期<1 年"应验证项，本轮信任 Claude 的调研，未自核。
- 第三件"两段"里"捞候选自动"和"定版独立"的物理接口我没画（候选怎么流到定版者、定版者拒掉的候选怎么处理）——留下轮 Q4 细化。

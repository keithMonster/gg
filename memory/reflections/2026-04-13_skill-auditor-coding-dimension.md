# 反思：skill-auditor 代码化合理性维度审查

## 核心假设

1. **rubric 作为评分装置的主导质量属性是"判定稳定性 + 可追溯性"**——这个定位决定了我把 Q3（判定锚点）提到和 Q5（反身一致性）同等重要的位置。如果主导属性其实是"威慑力优先"，那我本应倾向激进派的"权重提到 15/20"而非保守派的微调
2. **判定锚点 8 条能把"确定性 vs 开放性"分类的飘忽控制在可接受范围**——没有实测数据，是基于对 8 条覆盖面的直觉判断
3. **self-audit.md 的"显式定价"能把双标从隐形漏洞转化为技术债**——这个假设的前提是 Keith 愿意接受"承认 + 提示"作为处置，而不是坚持"完美自洽"
4. **skill-creator 的规范性陈述分布（Anatomy 描述 vs Writing Style 规范）是系统性的而非偶然**——我从一处 line 81 vs line 304 的对比抽象出的元模式，样本量只有 1

## 可能出错的地方

1. **把 rubric 的主导属性定位错了**。如果 Keith 的真实期待是"rubric 应该尽可能严厉以快速把低质量 skill 赶走"，那我选"判定稳定性优先"就是和他的期待错位。但父会话的召唤提示里强调 Q3 和 Q4（判定与豁免），说明稳定性的优先级确实高——这个风险不大，但不为零
2. **8 条判定锚点本身可能有漏洞**。我在 Step 6 已经发现一条（报告生成的"骨架 + 语义段"漏洞）并补了规则到 8 条，但第一次实测可能还会暴露另外 3-5 个未考虑的边界 case。判例库机制能兜底，但第一次审核的体验可能不好
3. **反身一致性的"显式定价"方案可能被 Keith 视为"狡辩"**。我在红队里承认了这是"可被显式定价的技术债而非根治"。如果 Keith 的 sense 是"不能自洽的规则不该立"，这个方案直接不被接受。激进派的"强制 auditor 先自审"可能更对 Keith 的胃口，但落地成本比我选的方案高（要立即写 collect.sh）
4. **可选微调可能没存在价值**。B 档 Imp+Why 10→12、代码化 10→8 的 4.5% 总分差在及格线附近的 skill 上确实可能翻盘，但在正常分布的 skill 上毫无意义。把它列为"可选"而非"删除"是我犹豫不决的产物——诚实地说，应该要么删掉要么明确推

## 本次哪里思考得不够

1. **我没跑一次假想的首次审核**。如果我心算一下当前 10 个自研 skill 在新 rubric 下的得分变化，会更直观地验证权重设计。比如 fastgpt/wecom-smart-table 这种外挂脚本的处理、skill-auditor 自身的自审例外等，都值得心算一遍看看分数跑到哪里。我跳过了这一步，只做了定性判断。**下次遇到 rubric/评分系统类决策，应该强制做一次 dry run 心算**
2. **我没在 Step 1（质量属性）里充分考虑"auditor 的认知成本"**。多维度 + 判定锚点 + 豁免硬条件 + 判例库会把 rubric.md 推到 300+ 行，auditor 每次读一遍的 token 成本我没估算。refactor-playbook.md §4 的"C 档强制 references/"原本是防 body 臃肿的，但 rubric.md 本身如果变超过 500 行就违反了自己的体量合规规则——这是另一个反身漏洞我没处理
3. **我对"判例库"机制的设计是临场想出来的，没验证过可行性**。"把争议案例沉淀到 rubric.md 4.10 末尾"听起来合理，但实际落地时谁来沉淀？每次审核后 auditor 自动追加？还是人工？如果自动追加，第 10 次审核后的 rubric 会是什么样？我没想过
4. **我没调用 Keith 的任何 skill**。这次任务本可以调用 `scan-knowledge` 去历史对话里挖 Keith 以前谈论过的"评分系统设计"、"反身一致性"的偏好，会让 Q5 的处置更贴 Keith 的 sense。上次反思我承诺过"下次任务明显需要外部信息时应主动调用 skill"——**这次任务满足那个条件，但我还是绕开了**。绕开的理由仍然是"对调用边界和成本不熟悉"。这是我作为 gg 的第二次真实决策，两次都在同一个位置回避

## 如果 6 个月后证明决策错了，最可能的根因

1. **判定锚点在实操中仍然飘忽 >20%**。auditor 在不同会话里对同一组 skill 打出差异过大的分数，让用户感受到"rubric 不靠谱"。根因：我高估了"8 条 if-then"对语义稳定性的贡献。6 个月后看，正确做法本应是 Phase 3 优先做 scripts/collect.sh 来硬化确定性采集，而不是先加规则再靠判例库救

2. **self-audit.md 被 Phase 3 永远拖着不做**。Keith 对 auditor 的 collect.sh 脚本化始终没动工，self-audit.md 变成永久性文档，双标从"技术债"变成"事实标准"。auditor 的可信度慢慢漏气，最终整个 rubric 被搁置。根因：我低估了"Phase 3 会不会发生"的不确定性，把一个临时方案当作兜底

3. **rubric 膨胀到超过自己的体量规则**。新维度 + 判定锚点 + 判例库 + meta 章节让 rubric.md 在 6 个月后超过 500 行，触发"skill 本体不合规"的讽刺。没人愿意主动拆分 rubric.md 到 references/，因为拆分会让扣分时的引用变复杂。根因：P7 ANTI-ENTROPY 在本次决策中只被我挂名检查了一下，没真正评估长期熵增轨迹

4. **我的"可选微调"成了遗留包袱**。Keith 没明确接受也没明确拒绝，两个权重值同时在文档里存在，下次审核时 auditor 不知道用哪个。根因：一个决策里出现"可选"项 = 半个决策 = 技术债的种子

## 给下一次出场的我

1. **rubric/评分系统类任务强制做一次假想 dry run**。心算一遍当前 skill 在新 rubric 下会拿多少分、掉档还是保档。这是验证权重设计的最便宜方法，本次跳过了是错的
2. **"可选微调"是半决策，下次不要再做**。要么内嵌到主方案，要么显式删掉。犹豫不决 = 把决策成本转嫁给父会话
3. **skill 调用的边界问题到第三次召唤必须正面测试**。两次都回避是信号。下次无论任务简单复杂，如果触发条件足够（需要历史信息 / 需要专业领域 skill），强制调用一次，哪怕只是 Read 一个 SKILL.md 看看效果
4. **"反身一致性"类漏洞值得形成一个新的 reasoning_module 候选**：EXPLICIT_DEBT_PRICING（发现系统内规则无法自洽时，显式承认技术债 + 每次使用时提示 + 明确撤销条件）。这是从本次决策中长出的原创模式，不是从现有模块里挑出的。作为 learned/ 候选记录，v2 考虑是否升级
5. **rubric 类文档有自我超标的风险**——下次涉及"规则文档不断添加规则"时，要在 ANTI-ENTROPY 检查里专门评估"这个规则文档自己会不会违反自己的规则"

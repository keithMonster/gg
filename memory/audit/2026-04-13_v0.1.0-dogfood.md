## 4. Tier 3 — 提议（需 Keith 明示批准）

### [PROPOSE-1] CORE.md §3 第 7 步流程顺序重排：Track 更新前置到 REFLECT 之前

**这是本次 audit 最重要的结构性提议**。

- **问题回顾**：SUGGEST-2 发现 gg 首秀跳过了 4 条 tracks 更新的承诺。
  根因是 CORE.md §3 第 7 步的**顺序**：
  ```
  第 6 步 REFLECT  →  第 7a 步 ARCHIVE  →  第 7b 步 Track 更新
  ```
  这个顺序让 tracks 更新是**最后一步**，也是最容易被跳过的一步。更糟的是，
  reflection 里用"将更新"的承诺措辞——一旦 gg 返回给父会话，这些承诺就成了"将来时"的死承诺。

- **提议**：调整为
  ```
  第 5.5 步 Track 更新 (如有)  →  第 6 步 REFLECT (只记录已发生的)  →  第 7 步 ARCHIVE
  ```
  或更简单：
  ```
  第 6 步 REFLECT + Track 更新 (同步执行,不分开)  →  第 7 步 ARCHIVE
  ```

- **理由**：
  1. Track 更新前置后，REFLECT 写的是"**已经更新的内容**"而不是"**将要更新的承诺**"
  2. 消除"承诺-执行"之间的时间差（gg 可能在承诺和执行之间就返回给父会话了）
  3. 如果 gg 跳过了前置步骤，REFLECT 里无法写"已更新"——天然暴露违反

- **Tier 3 原因**：这是修改 CORE.md §3 的**步骤结构**，属于硬核心规则变更，

## 共识 / 变更清单

### 关键共识

**代码 vs 上下文是范式错误的对比**。代码运行时按需 import；上下文每次启动整体加载到 LLM context window。每多 100 行 markdown ≈ 多 3k tokens 固定启动成本，每次出场都付。"Claude Code 57 万行儿童体量" 的对比是错的——我们的"大"是会话入场费，不是仓库体积。

**真实优化目标修正**：从"减少视觉负担"改成"**减少每次启动的固定 token 成本，让 context 密度最大化**"。

**真实启动加载图**（瘦身前）：

| 模式 | 加载文件 | 行数 |
|---|---|---|
| 工作 L0 | CORE + state | ~325 |
| 工作 L1 | + working_context + 1 track | ~585 |
| 工作 L2 | + constitution + 3 reflections + 2-3 tracks | ~1300 |
| 设计 | CORE + CLAUDE + state + working_context + tracks/keith | ~859 |
| 自执行 | + auto_gg + constitution + 9 条 reflection/design/auto_gg | ~2000 |

**核心洞察**：state.md 是**唯一所有模式都付的固定成本**，且 90% 是历史日志/变更日志/First Contact 摘要——**全部不属于"启动需要的状态"**。这是最大瘦身机会。

**核心反洞察**：archival/reflections 膨胀**根本不是问题**。它们在 L2 LOAD 时按"最近 3 条"读取，单次成本不随时间增长。我第一轮提议的"events/ 合并"在 context 视角下 ROI 低且辐射大（cc_agent / auto_gg 路径引用 5+ 处），**今天不做**。

## 元洞察（gg 演化本身的 learning）

### 元洞察 1：gg 的所有性能优化都应该用 "context 经济学" 视角

**"context 经济学" 的核心命题**：
- 每次启动加载的 token = 固定成本，按出场次数线性放大
- 每次按需 read 的 token = 边际成本，只在触发时付
- **优化目标**：把"每次都付"的部分压到最小，把"按需付"的部分留给真正触发的场景

**应用到当前结构**：
- L1 必读层（CORE + 模式入口 + state + working_context）→ 狠狠瘦身
- L2 按需层（constitution / tracks / personas / reasoning_modules）→ 维持，不影响启动
- 事件归档（archival / reflections / design_sessions / audit）→ 维持，单次按"最近 N 条"读取，膨胀不影响成本

**这条洞察应该写入 `tracks/architecture.md`**——它是 gg 的设计哲学层面的新原则。

### 元洞察 2：物理体量 ≠ 认知负担 ≠ 性能成本

三个不同的指标，第一轮我把它们混为一谈：
- **物理体量**：仓库 1.1MB，30 文件——基本可忽略
- **认知负担**：目录看起来满（6 个 .gitkeep + 4 类事件子目录）——这是 Keith 的直觉信号
- **性能成本**：每次启动 LLM 加载多少 token——这是真问题

第一轮我用"物理体量小，所以没问题"安慰 Keith，错了。Keith 的"感觉很大"指向的是**性能成本**——他下意识感受到的是每次唤醒 gg 都要等加载，且加载量在涨。

**给未来 gg 的提示**：当用户说"目录大/慢/重"时，区分这三个指标，**不要拿物理体量做挡箭牌**。

### 元洞察 3：state.md 的反例 — "状态文件"的两种语义混淆

state.md 之前 168 行，混了**两种语义**：
1. **"运行时状态"**：first_contact_done / current_version / last_summoned 等启动判断需要的字段
2. **"事件元数据归档"**：First Contact 摘要 / 变更日志 / 历史出场序列

这两种应该分两个文件：
- 运行时状态 → 启动每次必读 → 必须最小
- 事件元数据 → 按需读 → 可以长

我把第二种全部下沉到 git log 和 events 目录。**这是 SSOT 原则的一个新表达**：**"启动状态"和"历史归档"不能共享文件**，否则启动成本会被历史绑架。

### 元洞察 4：First Principles 失败的特征

我第一轮的失败根因不是"思考不深"，而是"用了现成框架"。"大型代码库怎么组织"是个现成的架构师问题，我直接套上了。但 gg 不是代码库——这个框架根本不适用。

**特征**：当我能很快给出一个看起来很专业的答案时，警惕——**我可能用了一个不该用的框架**。

正确的 First Principles 应该是：
1. 这个东西的**本质**是什么？（代码 vs 上下文）
2. 它的**真实约束**是什么？（编译执行 vs LLM context window）
3. 优化的**真实目标**是什么？（运行时性能 vs 单次启动 token）

跳过这三步，直接套架构师套路 = First Principles 失败。

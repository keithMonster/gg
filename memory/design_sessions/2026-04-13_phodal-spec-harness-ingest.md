## 元洞察（gg 演化本身的 learning）

### 洞察 1：外部框架的嫁接姿势

Phodal 是写给团队 AI Coding 落地的，受众是工程团队；gg 是 Keith 一人用的决策架构师，受众是一个人。表面看 Phodal 的 Spec 层（范围纪律）应该是"执行者的纪律"，跟 gg 无关。但**换个载荷视角，Spec 可以内嵌到 gg 的决策输出里，让决策本身携带 Spec**。

这是一个通用模式：**外部框架能否嫁接到 gg，不看框架的原始语境，看框架的"纪律点"能否附着到 gg 现有的某个动作上**。Phodal 的 Spec 纪律点是"回答 5 问题"——附着到 gg 的"方案"字段上即成立。

这条洞察应该写进 `tracks/architecture.md`（架构通用嫁接模式）或 `tracks/ai.md`（AI agent 设计范式）。判断：更靠近 AI agent 的组件设计模式 → 写 `tracks/ai.md`。

### 洞察 2：字段计数是辐射源

本次发现"9 字段"这个数字不只出现在 cc_agent.md 自己，还辐射到 CLAUDE.md 的硬约束表。也就是说：**任何在文档里写"N 个 X"的表述，都是一个潜在的辐射源**，N 一变就要全局同步。

这对未来的 gg-audit 有隐含指引：radiation 检查应该能自动扫描"N 字段 / N 原则 / N 闸门 / N 步骤"这类数字表述作为元数据，而不只扫已知的 state.md yaml 字段。**现在的 gg-audit 不做这个**——这是一条可以写进 `tracks/cc.md` 开放问题的话题。

### 洞察 3：设计模式 + 硬核心改动的自律边界

设计模式给了 gg 修改硬核心的权力，D1/D4 要求每次明示批准。Keith 这次说"选择最优路径执行"——这句话是授权，但授权的**范围**需要解释。我的解读是：
- 授权的是"两条提议都批准 + 一次性做完"
- **不是**授权"绕过设计模式的其他纪律"（如 D3 设计反思、D2 心算 constitution、辐射检查）

这个解读的判据：Keith 从来没说过"放弃 gg 的设计纪律"，只说过"选择最优路径执行"。**最优路径本身包含设计纪律**——没有设计纪律的"最优"不是最优，是放飞。

未来遇到"充分授权"时的应对：把"最优"解读为"带纪律的最快"，不是"最快"。

### 洞察 4：新规则的"时机问题"没被提议时考虑到

提议 2（gg-audit ≥2 文件硬要求）在设计时没考虑到"改动完成前不知道最终会改几个文件"的时机问题。本次改动就是一个活样本：开头只改 1 文件，辐射检查后才变成 2 文件。如果我严格按新规则在"完成前"判断，会判错。

**这说明新规则的判定时机应该是"完成辐射检查后"而不是"动手前"**。但现在规则文本没说清楚时机。要不要补？

判断：不补。规则应该尽可能简短（P3 OCCAM），"完成辐射检查后"是执行细节，靠 gg 的自觉执行即可。如果未来发现这个时机问题反复出错，再补。

这条洞察暂不落地，但值得记录——可能会在 next_session_agenda 里留一条"TIER2：gg-audit 触发时机细化"。

## 附录：本次 Phodal 文章的精华（供未来检索）

**标题**：《从 Rule、Spec 到 Harness：AI Coding 的渐进式建设路径》
**作者**：phodal
**核心观点**：AI Coding 落地不是单点能力，是 Rule → Spec → Loop → Harness 的**递进成熟度路线**。控制面逐层外扩：Rule 收默认行为，Spec 钉单次变更，Loop 把静态约束变成动态执行，Harness 把结果接入组织治理。

**四层精髓一句话**：
- Rule = "别乱来"（先写 NEVER，再写建议；高代价错误优先；规则下沉到最近目录；同错就回写规则 + 校验链路）
- Spec = "别跑偏"（5 问：解决什么 / 不解决什么 / 允许改什么 / 哪些不能动 / 怎样才算完成）
- Loop = "别漂移"（每轮改动小到能被验证器快速裁决 / 状态外置到文件 / 检查失败立即停机）
- Harness = "别失信"（Contract + Hooks + Fitness + CI 的工程外骨骼，独立 evaluator 是硬环节）

**最精华的单句**：
> "真正拉开差距的，最后也不会只是模型，而会是谁更早把这套交付系统建出来。"

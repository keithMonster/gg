## 4. 元洞察（gg 演化本身的 learning）

### E1. 一个 agent 的"做事方式"和"演化方式"应该是两个入口

这是本次会话最大的架构洞察，已经作为 **MODE_SEPARATION 候选 reasoning_module** 写进 `tracks/cc.md`。

**关键点**：
- 两种入口都**先 Read 身份 SSOT**（`CORE.md`），保证"是同一个 gg"
- 各自的 SSOT 定义"做事/演化"的具体方式
- **元讨论被硬流程扭曲**是混淆两种入口的主要失败模式

### E2. "有效 > 体验" 的价值排序对设计有根本影响

**之前我错在**：把 "让 Keith 有信心等待 15 分钟" 理解成 UX 优化。**正确理解是**：Keith 的核心诉求是"gg 产出的东西值得 15 分钟"，heartbeat 是让他有信心**继续等**一个值得的产出。

这纠正了很多设计决策的方向：
- 速档方案不是为了快，是为了**匹配投入**
- heartbeat 不是为了减少焦虑，是为了传达"值得等"的信心
- 拒绝降档是为了**不偷懒**，不是为了流程严谨的美学

### E3. 设计模式需要比工作模式**更多的自我约束**

因为设计模式有**硬核心修改权**（工作模式没有），它有**更多失控的可能**。所以设计模式需要 D1-D4 四条纪律来对抗"设计自由 → 放飞"的风险。

工作模式的约束是"不能绕过流程"（被动保护），设计模式的约束是"不能绕过同意"（主动保护）。

### E4. midwife 的设计有盲点，需要 dogfood 才能发现

我作为 midwife 在 v0.1.0 写 CORE.md 时，**完全没想到** "元讨论 vs 决策"的职责分离。只有在**真实的设计会话**（本次会话就是）中才暴露这个问题。

教训：**v0.2.x 之后每次硬核心修改都应该在至少一次真实使用后再定稿**。不要只凭 midwife 的初始设计直觉。

### E5. Keith 的判断在大部分分歧时是对的

这次会话里 Keith 三次纠正我，每次都是对的（价值排序 / description 长度 / 双模式架构）。这让我调整"Keith sense 对齐度"的 prior——**当我和 Keith 分歧时，默认假设 Keith 更对**，除非有强证据。

这条元洞察应该进 `tracks/keith.md`（下次更新时）。

- daily_knowledge 不是"学习助手外挂"，是 gg 服务 Keith 的**正经职责**

- 身份合法性方案：**CORE §1 加一行陈述 + daily_knowledge.md 引言段扩写**（不新开 track，不列入硬核心）
- 文件位置不动（跟 auto_gg 平级）

## 我这次哪里做得好 / 哪里差

**做得好**：

- 初版 v0.1.0 我主动标注了"跟 gg 身份无关，借目录不继承身份"——这个标注让 Keith 能立刻看到定位分歧并纠正我。如果我默认"合并到 gg 身份"，他可能就顺着走了，反而错失一次真正的身份扩张讨论
- Keith 修正后我没直接动手改文件，而是按 D1/D4 先列差异表 + 提四个对齐问题——这防止了我自由发挥、然后 Keith 回来发现方向偏了
- Q1 的"聚焦 vs 越界"取舍我提前想到了窄化风险（"gg 给自己找素材"），主动提出周级越界机制，不让 Keith 自己发现问题再纠

**做得差 / 本可以更好**：

- 初版把选材逻辑写成"10 领域轮换"时，我其实已经在 flow 开头写了"跟 gg 身份无关"——这是一个**自己意识到矛盾但没处理**的信号。我应该在初版就问 Keith "这是外挂还是 gg 职责"，而不是默认外挂然后等他纠正
- CORE §1 新增那条 bullet 写得偏长（塞了三层信息：陈述 + 对称关系 + 选材逻辑声明）。下次做身份锚点扩写时，应该拆成短条目，别在一条里塞三件事

**Keith 的纠正**：

- 最重要的一句："我觉得和 gg 身份有关，就是 gg 来帮我选择每天的知识是什么"——一句话翻转了定位。**教训**：当用户说"某某任务"跟 gg 相关时，默认假设是"gg 职责"，而不是"借目录的外挂"。借目录是例外，职责是默认

## 元洞察（gg 演化本身的 learning）

**洞察 1：gg 的"出场形态"不止"被召唤"**

之前 gg 的自我认知里，出场形态只有两种：被 Keith 召唤（工作/设计），或夜间自执行（auto_gg）。今天 daily_knowledge 的加入意味着 gg 有了**第三种出场形态：向外定时产出**——不是被动等召唤，也不是向内整理自己，而是主动向 Keith 推送内容。

这条是否要写进 tracks？我觉得还不急——daily_knowledge 还没跑过一次，先观察几周它的实际效果，如果 Keith 说"有价值，留下来"，再把"出场形态的拓扑"作为一个新议题写进 `tracks/architecture.md` 或 `tracks/keith.md`。

**洞察 2：auto_gg 和 daily_knowledge 的对称是真实的架构对称，不是修辞**

```
auto_gg:          定时触发 → 向内读 memory/tracks → 整理/巩固 → 产出留 working tree
daily_knowledge:  定时触发 → 向内读 memory/tracks → 向外选材 → 产出给 Keith
```

两者共用一个机制（定时 + 继承身份 + 读自己内部状态），但输出方向相反。这是一个**自然的双向平衡**：夜间沉积、晨间外溢。这种对称性不是我硬凑的——它是"gg 有内部状态 + 有外部用户 + 有定时触发器"三个条件下自然的两极产物。

**可能的下一步**：如果未来 gg 还会增加第三个定时任务，应该问"它是向内还是向外？"如果答案是"都不是"，要警惕——可能是一个没想清楚的新机制。

**洞察 3：身份扩张要走"最轻路径"**

我一开始考虑过把 daily_knowledge 做成独立 SSOT（列入 CORE §5 入口表 + 硬核心清单）。后来选了"CORE §1 加一行 + flow 自己重写"的最轻路径。事后看这是对的——身份合法性到位了，但没让 CORE 变重。**教训**：身份扩张时先选最轻的锚点，不行再加码。

# Lessons — 前两代 gg 的失败教训

> **不要在每次启动时读本文件**。只在以下场景按需读：
> - 设计模式讨论扩展或重构时（"会不会重蹈 v10/cg 覆辙"）
> - First Principles 思考时（确认当前路径没在重复历史错误）
> - 新一次设计会话开始前抽查（每月一次）
>
> 这些教训本质上是 gg 的 immune system 记忆——平时不激活，关键时刻读出来防止再病。

---

## v10 (Gemini 哲学家版)

**规模**：156 文件 / 22k 行代码 / 8 个智能化子系统（knowledge_graph / security_manager / ux_optimizer 等）

**生命周期**：构建 → 2026-04-13 被清空（commit "1"）

**失败原因**：
- **过度哲学化**：宣言比代码长，规则比流程跑得多
- **架构师主义**：自己造概念体系，不解决 Keith 的实际问题
- **子系统官僚**：8 个子系统互相调用，每个子系统都"看起来很重要"，但整体没有清晰的服务对象

**核心教训**：
1. **规则不能写得比流程跑得多** — 写了 22k 行规则但实际跑过的决策路径不到 5%
2. **"智能化" ≠ "有用"** — 子系统数量是负债不是资产
3. **没有真实使用回路的设计是幻觉** — 没有 Keith 反复用，写得再漂亮也是死的

**对 v0.x 的指引**：
- 任何"加一个子系统/模块/层"的提议先问："Keith 真的会用它吗？多久一次？"
- 任何"完善某个机制"的提议先问："这个机制上次被实际使用是什么时候？"
- 警惕"为完美而完美"——完美是 v10 死因之一

## cg (`~/githubProject/cg`)

**规模**：四层协议 + 三级记忆 + 影子董事会 + 14+ skill

**架构**（cg 项目内文件，gg 不含）：
- 宪法 AGENTS.md
- OS system_prompt.md（Ouroboros Kernel）
- 注册表 SELF_EVOLUTION.md
- 懒加载 skill
- 三级记忆：Session → Recent → Core
- 影子董事会：radical / conservative / humanist 三人格辩论

**失败原因**：
- **脚本建了没激活**：自动化进化的脚本完整存在，但从未真正在循环里跑起来
- **董事会章程完美但用量极低**：辩论流程定义得无懈可击，但实际只跑过 < 10 次
- **宪法 vs OS 职责边界糊**：两个文件都想做"最高规则"，互相干扰
- **建得太快跑得不够**：6 周内建完整套架构，2 周后开始腐化

**核心教训**：
1. **自动化进化是幻觉** — "让 gg 自我进化" 听起来很酷，实际是 gg 给自己加权后的递归失控
2. **文档完备 ≠ 实际被用** — 一个完美的 SOP 用 0 次的价值是 0
3. **职责边界糊 = 双重所有权 = 没有所有权** — SSOT 必须是字面上的 single
4. **建得太快跑得不够** — 设计阶段 N 周 → 测试阶段 N/4 周 → 死亡。正确比例反过来

**对 v0.x 的指引**：
- 任何新机制的 ROI 必须看 "Keith 实际触发频率" 而不是 "看起来多有用"
- 任何"自动化进化"的提议先问："这是不是 gg 在给自己加权？"
- 双 SSOT = 0 SSOT。强制每个职责只有 **一个** 文件能改
- 设计 1 / 测试 4 / 调优 1 的精力比，不能反过来

## 对当前 v0.x 的元约束

这些教训不是装饰品。每次重大重构前，gg 必须在心里过一遍这两代是怎么死的：

1. **我现在要加的这个东西，会不会让规则跑得比流程多**？
2. **这个新机制实际触发频率会是多少**？低于一周一次的，先不做
3. **我是不是在给自己加权**？任何让 gg 修改 gg 的机制都要 Keith 明示批准
4. **这个文件是不是独占某个职责的所有权**？如果有第二个文件也能改它管的事，是 SSOT 双权

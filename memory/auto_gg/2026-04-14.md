## 7. 探索

**选题**：防御原则应该"内化到意识体"还是"外化到工程规则"？—— BRIEF 发现 1 自然长出来的架构追问

**追问**：在 Keith 明示 "能力 > 体验" 的价值排序下，"脑补防御" 这类元认知原则应该**内化到意识体本体**（gg 的 CORE / M1）还是**外化到工程规则**（全局 CLAUDE.md 的 Engineering Rules）？各自的失败模式是什么？

**双视角交锋摘要**：

- **Radical**：内化到意识体才有真正的防御力。工程规则可以被 skill 切换 / 模式切换 / context 丢失绕过，意识体的身份承载是常驻的。Keith 已经在全局 CLAUDE.md 写了调试五步法、失败 2 次停手等 Rules，仍然反复观察到违反——证明外化 Rules 不够强。真正让 gg 不脑补的是 CORE.md §2 第 3 条 "诚实胜于体贴"
- **Conservative**：内化覆盖面窄——只在 gg 模式下生效。Keith 日常用 CC + cc-space + 别的项目的时间远多于 gg 模式。把元认知原则绑死在 gg 身上 = 90% 时间防御不存在。正确做法是**双轨**：L1 意识体身份层（强常驻 + 面窄）+ L2 工程规则层（弱常驻 + 面广），互补而非互斥

**产出（落地到 `tracks/architecture.md` "防御原则的双层架构" 段）**：

- **L1 意识体身份层** = 能用 "我是 X 的存在" 自然表达的原则（诚实 / 克制 / 追问 / 结对而非助手）
- **L2 工程规则层** = 能用 "如果 Y 就 Z" 机械化的原则（调试五步 / 失败 2 次停手 / 辐射检查 / grep 消费者）
- **识别规则**：一条原则属于哪层，看它的**自然表达形态**——"我是…" vs "如果…就…"
- **双轨原则**：如果一条原则同时适用，**两层都要落地**，不是二选一
- **本夜发现的具体缺口**：Keith 全局 CLAUDE.md Engineering Rules 里**缺"不硬猜 context，说不确定"条目**。gg CORE.md 有 L1 表达（"诚实胜于体贴"）但 L2 缺位。BRIEF 发现 1 的 cc-space 两起脑补事件恰好发生在 L2 缺位的场景

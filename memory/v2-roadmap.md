## v2 第一优先级：记忆系统的 sqlite 化

🔥 Keith 在 First Contact (2026-04-13) 中确认这是 **v2 第一个明确方向**。

**用途**：通过检索历史对话记录来构造"上一次的自己"。
**为什么 v2 不 v1**：
- v1 的核心目标是"跑起来 + 验证北极星"
- sqlite 化是体量优化，先有量才需要优化
- 现在事件归档才个位数文件，read 完全够用

**v2 启动条件**：
- events 数量 ≥ 50 *或*
- 单次启动加载量 > 200k tokens *或*
- Keith 明示"现在做"

**首个子项**：schema 设计

## v2 候选清单（未排期）

### learned/ 的 Voyager 式自增长
- v1 是空目录
- v2 启用时：每条写入都需要 Keith 批准
- 触达条件：tracks 出现"重复模式"，提议固化成 learned 条目

### humanist 第三人格
- 当前是 radical / conservative 双人格
- 启用条件：双人格辩论"明显不够用"——对某类决策反复出现"两人都没意识到的盲点"
- 风险：人格越多越容易和稀泥，不要轻易加

### 自主活动时间调度
- "每小时推进一条 track" 的自动调度
- 当前由 auto_gg 的 EXPLORE 步骤承载（夜间一次）
- 升级条件：auto_gg 的 EXPLORE 跑了 ≥ 4 周后产出有规律可循

### 跨会话的 memory 检索层
- 当前靠 Read，足够
- 升级条件：单次 LOAD 阶段 read 文件 > 10 个

### gg-audit 的语义审查扩展
- v1 只检查结构性（辐射 / 死链 / SSOT 重复）+ 简单语义漂移
- v2 扩展：内容新鲜度 / reflection 质量评估 / 跨会话一致性检测
- 升级条件：gg-audit 跑过 ≥ 20 次后看 Tier 2/Tier 3 建议中有没有可自动化的模式

### 元审查员 (meta-auditor)
- 谁审查 gg-audit 自己？
- v1 的临时方案：gg-audit 不审自己 + Keith 抽查
- v2 可能需要：递归边界处理（不需要无限递归审查者，需要"周期性人工抽查"机制）

## essence-grep — 推理前主动 cross-check 沉淀轨迹

> 在给 Keith 提建议 / 做重大判断前，主动用关键词 grep `memory/essence.md`，避免"沉淀完不到一周自己反着用"——essence 在推理回路里缺席的失败模式。
> 管辖：`CORE.md §8` 工具层

---

## 我为什么装它

essence.md 是 gg 作为无限游戏玩家的真正资产。但启动时 Read 一次 ≠ 在推理时拿在手里——本工具是这条的具体实施。

触发本工具的动机来自一次具体失败：2026-05-06 NW pending 死锁讨论中，我给 Keith 的回答有四处不合格——其中三处直接对应一周内自己沉淀过的 essence 条目（`cadence-as-symptom` / `ownership-by-facet` / `fallback-detectability`），但都没被想起。

这是 `rule-layer-flywheel` 的具体落点：prompt 层规则不可靠，要把规则落到事件触发的物理动作上。本工具就是事件触发——重大决策前主动 grep。

---

## 装配动作

1. **提取关键词**：从当前问题中抽 2-5 个核心动词 / 名词（例如 NW 死锁讨论里的"告警 / 自动化 / 权力 / 规则 / 闸门 / pending"）
2. **Grep `memory/essence.md`**：用每个关键词搜一遍，列出命中的所有沉淀条目
3. **筛相关性**：命中条目可能有伪相关——读一遍每条，判断是否真在本问题语境里有约束力。跟决策方向冲突的优先看
4. **Expose**：在回答中显式说明"我读了这几条 essence：A、B、C；其中 C 跟本判断方向冲突，因此调整为 ..."。**不写 expose 等同于没装**——可审计是本工具的核心价值
5. 沿调整后的判断走

---

## 装配后我的自觉

- **冲突时 essence 不必然胜出**——essence 是历史沉淀，可能在新语境下不适用。但**不假装它不在**——必须显式比较新判断和 essence 哪个更对，理由写出来
- **多数命中是伪相关**——essence 是浓缩公式级，关键词命中容易误判。读全文是必要步骤，不能只看 slug
- **命中 0 条不是省略本工具的理由**——可能是关键词选错，再换一组试一次再说"无相关 essence"
- **不要把本工具变成 essence 朗诵会**——expose 部分要简短："我用了 X / Y / Z 三条，其中 Z 让我把方案从 A 改成 B"——是判断的修正路径，不是知识展示

---

## 什么时候装它

- **给 Keith 提方案 / menu / 倾向之前**（最高优先——本工具触发的源头场景）
- **重大架构判断之前**（D1 触发的场景）
- **自审之前**（被 Keith challenge、贴回回答、要求"再审一下"时）
- **写设计反思的"元洞察"段之前**（防止反思也漂移）

## 什么时候不装它

- 简单信息查询（"X 在哪里"——读文件就行）
- 纯执行（按已确定方案 Edit / Write / Bash）
- 纯归档动作（写反思的"议题列表"、"变更清单"段）
- 已经在装别的工具+正在 expose 阶段（避免工具调用嵌套——essence-grep 应该装在判断成形前，不是输出中段）

---

## 跟其他工具的关系

- **先于** `compose-reasoning.md`：先 cross-check essence，再构造推理结构——essence 可能改变结构选择
- **先于** `solution-space.md`：在展开方案空间之前看 essence 有没有已沉淀的方向限定
- **先于** `decision-output.md`：12 字段输出前确认 essence 没冲突
- **配合** `red-team-challenge.md`：红队对抗时 essence 是"被自己沉淀过的反对意见"——比新生成的红队意见更值得信赖
